\section{Context}

Maths education is a crucial part of everyone's lives. Its primary goal is to equip students with fluency in the fundamentals of mathematics, mathematical reasoning, and problem-solving, and to apply mathematical concepts in science and other fields~\parencite{Math_Curriculum_England}. Government reports suggest that the average class size is around 26.2 pupils~\parencite{Class_Size_England} and has an average pupil-teacher ratio of 18.0~\parencite{Pupil_Teacher_Ratio_England}. This high ratio inevitably limits the attention a teacher can provide to each student.

Learning is separated into two groups: procedural fluency and contextual understanding. In maths, there is a large emphasis on procedural fluency in the early stages of education. In the UK, pupils do mental arithmetic in Key Stage 1. This is why calculators are only introduced at the end of Key Stage 2~\parencite{Math_Curriculum_England}. 

\subsection{Problem}

When teaching early arithmetic and multiplication tables in schools, the classical approach is to give pupils a long list of exercises. However, this approach is not optimal as a pupil can encounter problems that they have already mastered. Alternatively, they may encounter problems that they are unable to solve. A better approach is to keep the student in the Zone of Proximal development~\parencite{Zone_of_Proximal_Development}.

Keeping students in their Zone of Proximal Development is a known challenge, especially in large classes. A teacher can't adjust materials to each student individually. This is why teachers are forced to make trade-offs. To address this, I will use Knowledge Tracing. Knowledge tracing adjusts the materials based on a measured mastery of each topic. The mastery is the probability of the user answering correctly. This mastery is then used to estimate the Zone of Proximal Development. 

Because Knowledge Tracing algorithms create an identifiable state of mastery, it comes with data privacy concerns. Specifically, the General Data Protection Regulation complicates the collection of training and testing data. Existing papers show the possibility of avoiding these concerns by synthesising user data. These papers use publicly available student data and extend it using different synthesis methods~\parencite{pagonis2024studentsynthesis}. The challenge will be adapting these methodologies to basic maths and response time.

\section{Aim}

The overall aim of my project is to create an artifact of an education application that enhances learning using Knowledge Tracing algorithms. I intend to train and test the effectiveness of the Knowledge Tracing algorithms using synthesized student data. 

\subsection{Objectives}

\begin{enumerate}
    \item I will create a artifact that has a user interface (UI) for human interactions and an application programmable interface (API) for synthetic users. Both the UI and API should accept answer score and user response time, preprocess this data and send it to the Knowledge Tracing algorithms.
    \item The artifact should be able to handle multiple Knowledge Tracing algorithms. This should enable the comparison of multiple KT algorithms.
    \item I will implement a user synthesis system and expand it to generate user response times for answers. A working generator for synthetic user data should be able to arbitrarily many answer scores and response times.  
    \item The artifact will be evaluated on the synthetic student data. The knowledge tracing algorithms will be compared based on the synthetic user data as this method is controlled and repeatable.
\end{enumerate}

This initial artefact should work from start to finish, without having all the features. It should have a UI and a Knowledge Tracing algorithm. Around the mid-project check-in, I should have a working prototype and I should be preparing it for more features. Additionally, I hope to have a written the introduction section, and began working on the background research and literature review.

When I finish polishing the initial version of the artefact, I will begin implementing different Knowledge Tracing algorithms. At this stage I will also create user synthesis to test test the knowledge tracing algorithms. I implement these here, to have content for the poster due on the $20^{th}$ of March. 

During Easter, I intend to take a slight break and do some debugging and testing. Once I come back I intend to implement response time into both the knowledge tracing and the synthetic users. 

When I'm done with implementing the response time, I will test and evaluate both the knowledge tracing and the user synthesis. Once I'm done with these tests, I will have some time to finish writing my dissertation and polish the artifact ready for presenting during the Demonstration Fair. If I have some spare time at the end I intend to gamify the UI\@.

One at the beginning of Easter and one towards the end of the project.  
\end{itemize}
