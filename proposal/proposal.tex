\section{Context}

Maths education is a crucial part of everyone lives. It's primary goal is to equip students with a fluency in the fundamentals of mathematics, mathematical reasoning, problem-solving, and to apply mathematical concepts in science and other fields~\parencite{Math_Curriculum_England}. Government reporting suggest that the average class size is around 26.2 pupils~\parencite{Class_Size_England} and has an average pupil to teacher ratio of 18.0~\parencite{Pupil_Teacher_Ratio_England}. This high ratio inevitably limits the attention a teacher can provide to each student.

Learning is separated into two groups: procedural fluency and contextual understanding. In maths there is a large emphasis on procedural fluency in the early stages of education. For example, in the UK pupils written and mental arithmetic in key stage 1. In order to minimise reliance on calculators, and promoting a deeper understanding of maths, the national curriculum suggest calculators should be introduced in schools near the end of key stage 2~\parencite{Math_Curriculum_England}. 

\subsection{Problem}

When teaching early arithmetic and multiplication tables in schools, the classical approach is to give pupils a long list of exercises. However this approach is not optimal as a pupil can encounter problems that they have mastered already and find easy or alternatively encounter problems that they would struggle with. A better approach is to keep the student in the Zone of Proximal development~\parencite{Zone_of_Proximal_Development}.

One of the down sides of ZPD is that it's difficult to implement for large classes. Teachers cannot adjust the education materials to each student individually, and are forced to make trade-offs. I intend to use Knowledge Tracing to create real time adjusting of education materials with the aim of keeping students in their Zone of Proximal Development. Knowledge Tracing Algorithms work by estimating the mastery of Knowledge Components in real time. The mastery is expressed as a percentage of how likely a user is to answer correctly. This percentage is used to inform a problem selection algorithm.

Research into Knowledge Tracing Algorithms often comes with data privacy concerns. The General Data Protection Regulation makes it difficult to collect training and testing data, because Knowledge Tracing Algorithms create an identifiable state estimating mastery of skill. In my case additional concerns are raised as the users of my artifact would be children learning simple arithmetic (Key Stage 2 and bellow). Existing papers show the possibility of avoiding these concerns by synthesizing users data. These papers -use publicly available student data and extend it using different synthesis methods~\parencite{pagonis2024studentsynthesis}. The challenge will stem in adapting these methodologies to my use case.

\section{Aim}

The overall aim of my project is to create an artifact of an education application that enhances learning using knowledge tracing algorithms. I intend to train and test the effectiveness of the knowledge tracing algorithms using synthesized student data. 

\begin{enumerate}
    \item I will create a generator for synthetic student users. Synthetic students will allow me to measure and train the data. A successful synthetic user should be able to send the correct data to the game with the correct response time. 
    \item Create a interchangeable knowledge tracing algorithms that communicate both with the game and synthetic users. I will know this objective is satisfied once I've implemented at least two knowledge tracing algorithms that can be trained using synthetic users and are capable of sending problems to the unity game and receive user responses. 
    \item Create a Unity game prototype, with an a replaceable knowledge tracing backed. The working game should serve as a UI for the whole prototype. I will have achieved this goal once I have a working game prototype that allows the backend knowledge tracing algorithm to ask the user questions and receive their answer along with a response time. Users are able to answer questions and their response times are recorded. 
    \item Test the game using quantitative learning rates. The game should be tested both using synthetic and real users. Real users also need to leave feedback on gamification.
\end{enumerate}

\subsection{Risk}
\begin{itemize}
    \item A risk I have identified is the lack of research into synthesizing response time. If the synthesis of users cannot be completed within its allocated time, I will drop the response time from the project and only focus on the correctness of the responses.
    \item A second risk is that training multiple knowledge tracing algorithms could exceed the scope of this project. If the time for implementing knowledge tracing algorithms is exceeded I will drop the remaining knowledge tracing algorithms.
    \item Another risk is that I fall ill or am otherwise not able to work on the project. As a contingency the game will be developed iteratively meaning that a working version of the game should be developed as soon as possible and the rest of the game can be dropped.
\end{itemize}
