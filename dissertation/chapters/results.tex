\chapter{Results}

This chapter presents the smart tutor system and student synthesis pipeline. Next, it explains the metadata and metrics of the generated datasets, then outlines evaluation results for selected knowledge tracing models. Specifically, these models are tested on LLM-generated, ASSISTments2009, and real app data. The chapter then follows the research questions from the Aims and Objectives section, which are restated below for convenience:

\begin{enumerate}
    \item To what extent can knowledge tracing algorithms be effectively evaluated and compared using synthetic student data generated by large language models?
    \item Which knowledge tracing model demonstrates the strongest predictive performance when both trained and tested on LLM-generated synthetic student data within the smart tutor app?
    \item To what degree do evaluation results on synthetic student data align with results on real educational data, such as ASSISTments and real app users?
\end{enumerate}

\section{Smart Tutor App}

The smart tutor app's front end is implemented in Unity, providing the user interface for interacting with the system. The backend consists of server-side logic provided via a REST API, written in Python. The entire project is hosted on a personal Raspberry~Pi~5 a compact, single-board computer. Users can access the app through https://dissertation.jakubwj.com. The dissertation codebase, including server, experiment scripts, and LaTeX files, is available at the provided repositories.

\begin{itemize}
    \item \textbf{Dissertation backend and LaTeX} at \href{https://github.com/jakubwj2/dissertation}{github.com/jakubwj2/dissertation}. This repository contains the server implementation, experiment scripts, the dissertation and proposal \LaTeX{} files, and includes the pyKT fork as a submodule.
    \item \textbf{pyKT fork} at \href{https://github.com/jakubwj2/pykt-toolkit}{github.com/jakubwj2/pykt-toolkit}. The original repository can be found at \href{https://github.com/pykt-team/pykt-toolkit}{github.com/pykt-team/pykt-toolkit}.
    \item \textbf{Unity project} at \href{https://github.com/jakubwj2/SmartTutor}{github.com/jakubwj2/SmartTutor}. This includes all Unity assets. Building and running the client requires a local Unity installation; the dissertation submission includes only a WebGL build.
\end{itemize}

The tutor currently includes four question generators: addition, subtraction, multiplication, and division. Here, a question generator is a software component that, given a maximum operand argument specifying the largest allowable operand value, systematically generates arithmetic practice questions. For these experiments, the maximum operand for each generator was set to 9. This means the operands ranged from 0 to 9 for all operations except the second operand in division, which ranged from 1 to 9 to prevent division by zero. With these settings, the system can generate 390 unique questions across these four arithmetic operations. This number would increase by raising operand limits or adding new generator classes.

\subsection{Observed Behaviour}

Beyond overall performance, analysis of item-level correctness revealed systematic patterns in the LLM-generated responses. The values can be found in Table~\ref{tab:answer_accuracy}. Firstly, all models performed better on addition and multiplication than on subtraction and division.

All models achieved similar accuracy above 0.94 on addition. On multiplication, all except DeepSeek-R1~\autocite{deepseek-r1} exceeded 0.97, while DeepSeek-R1 reached 0.89. For subtraction and division, Llama3.2~\autocite{llama3}, Mistral~\autocite{mistral7b}, and Qwen2.5~\autocite{qwen2.5} ranged between 0.85 and 0.89. In contrast, Gemma3~\autocite{gemma_2025} scored 0.59, and DeepSeek-R1 just 0.46. On subtraction, Qwen2.5 produced exclusively correct answers. In comparison, Gemma3 dropped sharply to 0.62, Mistral slightly to 0.58, and Llama3.2 to 0.53. DeepSeek-R1 generated the most incorrect answers overall. Whereas Qwen2.5 generated mostly correct answers.

Other observed patterns were that the incorrect subtraction answers occurred mostly when the correct answer was negative. In general, the models generated more incorrect subtraction answers than they did for division. In division, correctness was lower; most incorrect answers occurred when the correct answer was a non-integer rational number.

\section{Testing}

All six users completed the task without any issues. During testing, however, only one abruption was caused by a power outage. Once power was restored, the server restarted within minutes.

Users tested the website and then completed the System Usability Scale (SUS)~\autocite{SUS} questionnaire. The SUS article included a Google Form for participants to submit. As shown in Table~\ref{tab:sus_components}, these are the form results. Using Formula~\ref{formula:sus_score}, the calculated SUS score was 81.25.
